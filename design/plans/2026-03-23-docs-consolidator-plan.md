**Goal:** Build a Claude Code skill that consolidates scattered documentation artifacts into a navigable hub at `docs/_hub/`, providing a single entry point with links, summaries, staleness analysis, and gap detection.

**Architecture:** Two skill files — `SKILL.md` (workflow definition with 4-phase pipeline: discover → analyze → detect gaps → generate) and `references/hub-templates.md` (output templates for all 6 hub files). **Tech Stack:** Claude Code skill (Markdown), Bash/Git for analysis, Glob/Grep/Read for discovery, Write for output.

# Documentation Consolidator

Consolidate all project documentation into a single navigable hub at `docs/_hub/`.

## What This Skill Does

- **Discovers** all documentation artifacts across the repo (codemaps, specs, backlogs, devlogs, READMEs, ADRs, infra configs)
- **Analyzes** staleness using git commit dates and identifies documentation coverage gaps
- **Generates** a navigable hub at `docs/_hub/` with links, summaries, and actionable gap analysis
- **Exception:** generates original content only for `infra-ops.md` (no existing tool covers infra/ops)

## What This Skill Does NOT Do

- Does NOT replace `doc-updater` (codemap generation)
- Does NOT replace `dev-backlog` (backlog capture)
- Does NOT replace `dev-progress` (progress tracking)
- Does NOT auto-invoke other tools — reports gaps and suggests commands
- Does NOT auto-commit — user reviews output and decides

## When to Use

- After extended development sessions when docs have sprawled
- When onboarding someone to a project and need a "start here"
- After running `doc-updater`, `dev-progress:init`, or `dev-backlog:add` multiple times
- Periodically as a documentation health check

## Critical Rules

1. **`docs/_hub/` is fully managed.** Wipe and regenerate on every run. Never incrementally update.
2. **Read-only toward existing docs.** Never modify artifacts outside `docs/_hub/`.
3. **Self-exclusion.** All discovery patterns MUST exclude `docs/_hub/**`.
4. **Generated marker.** Every output file starts with: `<!-- Generated by docs-consolidator. Do not edit — will be overwritten on next run. -->`
5. **Git-based staleness.** Use `git log` commit dates, never filesystem mtime.
6. **No auto-commit.** Generate files, tell the user what was generated, let them decide.
```

Read the current content of `~/.claude/skills/docs-consolidator/SKILL.md` first, then append the following to the end:

```markdown

### Step 4: Check Git Availability

1. Run: `git rev-parse --is-inside-work-tree 2>/dev/null`
2. If this fails or returns non-zero → set `GIT_AVAILABLE = false`. Staleness analysis will be skipped.
3. If successful → set `GIT_AVAILABLE = true`.

### Step 5: Analyze Staleness (skip if GIT_AVAILABLE is false)

For each discovered documentation artifact, determine if it is stale:

1. Get the artifact's last commit date:
   ```
   git log --follow --format=%ci -1 -- <artifact_path>
   ```
   If the file is untracked (no git history), record it as "uncommitted" and skip staleness check.

2. Determine the related source path using these rules:
   - **Codemaps** (`docs/CODEMAPS/<name>.md`): extract `<name>` from filename stem. Check if `src/<name>/` or `<name>/` exists on disk. If neither exists, fall back to `src/` or `lib/` and record as "approximate match".
   - **Design specs**: Read the first 50 lines. Extract paths that begin with `src/`, `lib/`, `services/`, `packages/`, `apps/`, or `modules/`. Ignore paths inside fenced code blocks (``` or ~~~), paths starting with `/` alone (URL routes), and paths containing `://`. Use extracted paths as related source paths. If none found, fall back to `src/` or `lib/`.
   - **READMEs**: use the directory containing the README.
   - **All other artifacts**: use `src/` or `lib/` as a broad signal. If neither `src/` nor `lib/` exists, skip staleness for this artifact and note "no standard source root found" in GAP-ANALYSIS.md.

3. Check for commits newer than the artifact:
   ```
   git log --oneline --since="<artifact_last_commit_date>" -- <related_source_path>
   ```
   If any commits are returned, mark the artifact as **stale** and record the commit count.

4. Collect all stale artifacts into a `STALE_ARTIFACTS` list with: path, last updated date, commit count since, related source path, and whether the match was approximate.

### Step 6: Detect Coverage Gaps

1. Identify all top-level source directories (directories containing source code files — `.ts`, `.js`, `.py`, `.go`, `.rs`, `.java`, `.rb`, etc.).
2. For each source directory, check if ANY documentation artifact references it (codemap, README inside it, design spec mentioning it).
3. Directories with zero documentation coverage go into `UNCOVERED_DIRS` list.
4. Calculate coverage percentage: `(covered_dirs / total_source_dirs) * 100`.

### Step 7: Detect Documentation Gaps

Check for these specific gaps and record each in `GAPS` list:

1. **Missing codemaps**: source directories with no corresponding `docs/CODEMAPS/<name>.md` → suggest "Run `/update-codemaps`"
2. **Missing CI/CD docs**: CI/CD config files exist (`CI_CD` is non-empty) but no documentation references them → suggest "Run `/docs-consolidator` again after documenting CI/CD"
3. **Missing deployment docs**: `Dockerfile*` or `k8s/**` or `helm/**` exist but no docs reference them → suggest "Document deployment process"
4. **Dead specs**: For each design spec, extract filesystem paths (same rules as Step 5.2 for specs). For each extracted path, check: does the parent directory exist on disk but the path itself does NOT? If so, flag as dead reference. Record: spec path, referenced path, issue description.
5. **Missing backlog**: no `dev-backlog/` directory → suggest "Use `/dev-backlog:add`"
6. **Missing progress tracker**: no `dev-progress/` directory → suggest "Use `/dev-progress:init`"
7. **Missing devlog**: no `DEVLOG.md` → suggest "Sessions captured via `devlog-session-end`"
```

# 11 — Advanced Configurations

## Prompt Templates and System Prompts

### Custom System Prompt Injection via Rules

Since rules are injected into the system prompt, they serve as the primary mechanism for custom system-level instructions:

```markdown
# ~/.gemini/GEMINI.md
# This file is ALWAYS injected into every conversation across all projects

# Global Agent Instructions

## Identity
You are a senior software engineer working on production systems.
Always write production-quality code with proper error handling,
logging, and testing.

## Response Style
- Be precise and avoid unnecessary verbosity
- Always explain WHY, not just WHAT
- Provide code examples with real values, not placeholders
- When uncertain, state your assumptions explicitly

## Quality Standards
- Code must compile/lint on first attempt
- All public functions must have documentation
- Every change must include relevant tests
- Follow existing patterns in the codebase

## Safety
- Never commit secrets, API keys, or credentials
- Never run destructive commands without confirmation
- Always create a backup branch before large refactors
```

### Agent Instructions via Skill Definitions

For task-specific instructions, use skills instead of rules to avoid context bloat:

```markdown
---
name: "Strict Code Review Mode"
description: "Activates a rigorous code review process with mandatory checklists for critical code paths."
---

# Strict Review Instructions

When reviewing code in this mode:

1. Check every function for:
   - Input validation at boundaries
   - Error handling on all failure paths
   - No implicit type coercions
   - Proper resource cleanup (DB connections, file handles)

2. For each file changed, verify:
   - Tests exist and cover the changes
   - No regressions in existing functionality
   - Performance implications documented

3. Output a structured review report (see template below)
```

## Memory Tuning

### Knowledge Item Management

```
~/.gemini/antigravity/knowledge/
├── <ki-id>/
│   ├── metadata.json          # Summary, timestamps, references
│   └── artifacts/
│       ├── analysis.md        # Distilled knowledge
│       └── patterns.md        # Identified patterns
```

KIs are generated by a Knowledge Subagent that processes conversations and distills reusable knowledge. To optimize:

1. **Keep conversations focused** — One topic per conversation produces cleaner KIs
2. **Name conversations clearly** — Titles help with KI discovery
3. **Reference prior work explicitly** — "Building on the work from the auth conversation"
4. **Review KI summaries** — At conversation start, check if relevant KIs exist

### Context Control Strategies

```
+------------------------------------------------------------------+
|              CONTEXT OPTIMIZATION TECHNIQUES                      |
|                                                                   |
|  1. SPLIT CONVERSATIONS                                           |
|     One conversation per feature/topic                            |
|     Prevents context pollution between unrelated tasks            |
|                                                                   |
|  2. USE GLOB-ACTIVATED RULES                                      |
|     Only load rules relevant to current file types                |
|     activation: glob_pattern with globs: ["**/*.py"]              |
|                                                                   |
|  3. SKILL LAZY LOADING                                            |
|     Skills only fully loaded when agent deems them relevant       |
|     Write precise descriptions to avoid false positives           |
|                                                                   |
|  4. MCP ON-DEMAND                                                 |
|     Only connect MCP servers needed for the current task          |
|     Disconnect unused servers to save context                     |
|                                                                   |
|  5. FILE EXCLUSION PATTERNS                                       |
|     Exclude node_modules, dist, build from indexing               |
|     Exclude large data files and binaries                         |
|                                                                   |
|  6. STRATEGIC FILE READING                                        |
|     Read specific line ranges instead of entire files             |
|     Use grep_search to find relevant sections first               |
+------------------------------------------------------------------+
```

## RAG for Codebases

### How Project Indexing Works

Antigravity creates a semantic index of your project:

1. **File scanning** — Walks the project tree (respecting `.gitignore`)
2. **Content extraction** — Parses code into meaningful chunks
3. **Embedding generation** — Creates vector embeddings for semantic search
4. **Index storage** — Local index for fast retrieval

### Optimizing the Knowledge Base

```json
// .vscode/settings.json
{
  "antigravity.indexing.exclude": [
    "**/node_modules/**",
    "**/dist/**",
    "**/build/**",
    "**/.git/**",
    "**/coverage/**",
    "**/*.min.js",
    "**/*.map",
    "**/vendor/**",
    "**/__pycache__/**",
    "**/*.pyc"
  ],
  "antigravity.indexing.include": [
    "**/*.ts",
    "**/*.tsx",
    "**/*.py",
    "**/*.go",
    "**/*.rs",
    "**/*.md",
    "**/*.yaml",
    "**/*.json"
  ]
}
```

## MCP Server Configuration

### Setting Up MCP Servers

MCP servers are configured in `mcp_config.json`:

```json
// ~/.gemini/mcp_config.json (global)
// or .vscode/mcp_config.json (workspace)
{
  "servers": {
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL": "postgresql://user:pass@localhost:5432/mydb"
      }
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    },
    "filesystem": {
      "command": "npx",
      "args": [
        "-y", "@modelcontextprotocol/server-filesystem",
        "--root", "/path/to/docs"
      ]
    }
  }
}
```

### Available MCP Servers

| Server | Purpose | Use Case |
|---|---|---|
| **PostgreSQL** | Database schema + query | Writing accurate SQL and migrations |
| **GitHub** | PRs, issues, branches | Managing code changes and reviews |
| **Google Cloud** | AlloyDB, BigQuery, Spanner | Cloud database operations |
| **Filesystem** | Additional file access | Reading external documentation |
| **Jira** | Tickets and requirements | Ticket-to-code workflows |
| **Slack** | Team communication | Receiving requirements, posting updates |

### Accessing MCP via UI

1. Open the Editor View
2. Click the **"..."** dropdown in the agent sidebar
3. Select **"MCP Store"**
4. Browse available servers or click **"Manage MCP Servers"** > **"View raw config"**
5. Edit `mcp_config.json` directly

## External Tool Integration

### Custom Scripts as Agent Tools

While agents can't install new OS-level tools, they can execute scripts:

```
.agents/scripts/
├── validate-schema.py      # Agent runs: python .agents/scripts/validate-schema.py
├── generate-docs.sh        # Agent runs: bash .agents/scripts/generate-docs.sh
├── check-dependencies.js   # Agent runs: node .agents/scripts/check-dependencies.js
└── lint-sql.sh             # Agent runs: bash .agents/scripts/lint-sql.sh
```

Reference these in skills or workflows:

```markdown
## After creating the migration, validate the schema:
\```bash
python .agents/scripts/validate-schema.py prisma/schema.prisma
\```
```

### Integrating with Existing CI/CD

Agents can trigger and monitor CI/CD pipelines:

```markdown
# .agents/workflows/trigger-ci.md
---
description: "Trigger CI pipeline and monitor results"
---

1. Push changes to feature branch
// turbo
2. Run: `git push origin HEAD`
3. Check CI status:
// turbo
4. Run: `gh run list --limit 1 --json status,conclusion`
5. If CI fails, read the logs:
// turbo
6. Run: `gh run view --log-failed`
7. Fix the issues and re-push
```

## Automation Pipelines

### Multi-Step Automation Example

```markdown
# .agents/workflows/full-release.md
---
description: "Full release pipeline from version bump to deployment"
---

1. Determine the next version based on conventional commits:
// turbo
2. Run: `npx standard-version --dry-run`
3. Review the changelog preview
4. Bump the version:
5. Run: `npx standard-version`
6. Build the project:
// turbo
7. Run: `npm run build`
8. Run all tests:
// turbo
9. Run: `npm run test`
10. Push with tags:
11. Run: `git push --follow-tags origin main`
12. Notify developer of release completion
```
